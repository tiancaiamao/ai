# Agent Harness Evolve - 方向论

## 核心原则

**先手动验证，再自动化**

- 每个步骤都要单独实现并验证
- 手动运行每个步骤，验证输出格式
- 确认符合预期后，才考虑脚本化
- 脚本化后仍要手动监督运行几轮
- 只有多次成功后才进入完全自动化

---

## 实施策略

### 阶段 1: 步骤验证（手动）

1. **步骤 1: 测试集选择**
   - 选择 15-20 个稳定任务
   - 每个任务运行 3 次，验证稳定性
   - 创建 `evolve-manifest.json`

2. **步骤 2: 跑一次测试集**
   - 运行完整测试集
   - 生成 `iteration-0.json`
   - 验证 trajectory 字段

3. **步骤 3: 提取 trajectory**
   - 提取失败任务的 trajectory
   - 标准化为 OpenAI messages 格式
   - 验证格式正确性

4. **步骤 4: Debugger 分析**
   - 使用 LLM 分析失败轨迹
   - 识别根本原因和风险
   - 验证分析质量

5. **步骤 5: Planner 优化**
   - 基于 debugger 分析生成配置
   - 明确修改原因和预期效果
   - 验证配置格式

6. **步骤 6: 重测试并验证**
   - 应用新配置，重新运行
   - 对比结果，决定接受/拒绝
   - 验证决策逻辑

### 阶段 2: 自动化脚本（手动监督）

1. 编写 `evolve-loop.sh` 脚本
2. 顺序执行步骤 2-6
3. 手动监督运行 3-5 轮
4. 发现问题，立即修复
5. 验证脚本稳定性

### 阶段 3: 完全自动化

1. 多轮成功运行
2. 决策逻辑可靠
3. 错误处理完善
4. 日志和监控完备
5. 可以无人值守运行

- **现象**: evolve_loop.sh 使用旧的 result 文件进行对比

- **解决**: 先修复问题 1，再重新运行

## 关键设计决策

### 为什么手动验证优先？

1. **发现隐藏问题**: 自动化脚本会掩盖细节，手动运行容易发现问题
2. **理解数据流**: 手动运行每个步骤，能清晰看到数据如何流动
3. **验证格式约定**: 确保每个步骤的输入/输出符合设计
4. **建立信心**: 手动验证通过后，才有信心自动化

### 为什么固定命名？

1. **可预测性**: 脚本能准确找到需要的文件
2. **避免选错**: 时间戳命名导致无法预测"最新"文件
3. **历史清晰**: 直接从文件名看迭代序号

### 为什么独立 artifacts 目录？

1. **完整上下文**: input + output 都在同一个目录
2. **易于复现**: 拷贝整个目录即可重试
3. **隔离性强**: 不同迭代互不影响

## 风险和缓解

### 风险 1: 某个步骤无法验证
- **缓解**: 该步骤可能是设计有问题，重新审视设计

### 风险 2: 自动化后不稳定
- **缓解**: 手动监督运行多轮，发现问题立即修复

### 风险 3: 性能提升不明显
- **缓解**: 这是正常的，evolve 需要时间积累经验

### 风险 4: 遇到未知问题
- **缓解**: 记录问题，分析根因，逐个解决
