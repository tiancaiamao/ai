# PR #67 Review: Convert poor fuzzy match from error to warning

## Summary

This PR attempts to change the behavior when the edit tool finds a "poor" fuzzy match (high edit distance score). Instead of failing entirely, the edit should proceed with a warning.

## Analysis

### Intent
When the fuzzy matcher finds text with a score > 10 (poor match), instead of returning an error and failing the edit, the code should:
1. Proceed with the edit using the poor match
2. Append a warning to the result

### Critical Bug Found

**The implementation does not work as intended.** The `findBestMatch` function still returns `fmt.Errorf(...)` for poor matches, not `*PoorMatchError`:

```go
// Current code in findBestMatch (lines 343-348):
if bestMatch.score > 10 {
    matchedContent := content[bestMatch.start:bestMatch.end]
    return bestMatch, fmt.Errorf("fuzzy match is poor...")  // ← Plain error, NOT *PoorMatchError
}
```

The type assertion in Execute will always fail:

```go
if poorErr, ok := err.(*PoorMatchError); ok {  // ← ALWAYS FALSE
```

**Result:** The edit will still fail on poor matches, not proceed with a warning.

### How to Fix

The `findBestMatch` function needs to return `*PoorMatchError` instead of `fmt.Errorf(...)`:

```go
if bestMatch.score > FuzzyThreshold {
    matchedContent := content[bestMatch.start:bestMatch.end]
    return bestMatch, &PoorMatchError{
        Match:   bestMatch,
        Message: fmt.Sprintf("fuzzy match is poor (score %d), found: %q", bestMatch.score, truncateString(matchedContent, 50)),
    }
}
```

### Secondary Issues

1. **Unused variables defined:**
   - `FuzzyMatchEnabled` is defined but never used
   - `FuzzyThreshold` is defined (good!) but `findBestMatch` still uses hardcoded `10`

2. **Nil check missing:**
   ```go
   if poorErr, ok := err.(*PoorMatchError); ok {
   ```
   This will panic if `err` is `nil`. Need to check `err != nil` first.

3. **Redundant warning append:**
   If the type assertion succeeded (with the fix), `err` would be set, so the second check would append the warning correctly. However, the logic flow is confusing.

## Recommendation

**Request changes.** The PR has a critical bug that prevents the intended behavior. The `findBestMatch` function must return `*PoorMatchError` type for the warning path to work.

### Suggested Implementation Order

1. Fix `findBestMatch` to return `*PoorMatchError`
2. Fix the nil check in Execute
3. Consider using `FuzzyThreshold` variable instead of hardcoded `10`
4. Remove or use `FuzzyMatchEnabled` variable

## What Works Correctly

- The concept of separating "no match found" (error) from "poor match found" (warning) is sound
- The `PoorMatchError` struct design is appropriate
- The warning appending logic in Execute is on the right track (once the bug is fixed)
